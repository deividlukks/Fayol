Relatório de Funcionalidades do Backend - Projeto Fayol
A API foi desenvolvida seguindo uma arquitetura moderna em um monorepo, utilizando NestJS com TypeScript, Prisma como ORM para o banco de dados PostgreSQL e um sistema de autenticação robusto baseado em JWT. Todas as funcionalidades críticas são cobertas por uma suíte de testes automatizados End-to-End.

1. Módulo de Autenticação (/auth)
Este módulo gerencia a segurança e o acesso à API.

Endpoint: POST /auth/login

Funcionalidade: Autentica um usuário existente.

Lógica: Aceita um campo login (que pode ser o email, cpf ou o número de telefone do usuário) e uma password. O sistema valida as credenciais contra os dados criptografados no banco de dados. Se a autenticação for bem-sucedida, retorna um Token de Acesso (JWT).

2. Módulo de Usuários (/users)
Gerencia o ciclo de vida completo dos usuários do sistema.

Endpoint: POST /users (Público)

Funcionalidade: Registra um novo usuário.

Lógica: Exige name, email (único e obrigatório), cpf (único), phone (único e obrigatório) e password. A senha é automaticamente criptografada (hashed) com bcrypt antes de ser salva. O sistema previne a criação de usuários com email ou telefone já existentes, retornando um erro 409 Conflict.

Endpoints: GET /users, GET /users/:id, PATCH /users/:id, DELETE /users/:id (Protegidos)

Funcionalidade: CRUD completo para gerenciamento de usuários por administradores ou pelo próprio usuário.

Lógica: Todas as rotas são protegidas e exigem um token JWT válido para acesso.

3. Módulo de Contas (/accounts)
Permite que os usuários gerenciem suas contas financeiras (ex: conta corrente, carteira).

Endpoints: POST, GET, GET /:id, PATCH /:id, DELETE /:id

Funcionalidade: CRUD completo para as contas.

Lógica: Todas as rotas são protegidas. Um usuário só pode criar, visualizar, atualizar ou remover suas próprias contas. O sistema impede a remoção de uma conta se ela possuir transações associadas, garantindo a integridade dos dados.

4. Módulo de Categorias e Subcategorias (/categories, /subcategories)
Oferece um sistema de categorização flexível para as transações.


Funcionalidade: O sistema conta com categorias e subcategorias padrão globais e permite que o usuário crie, edite e exclua suas próprias categorias personalizadas.


Endpoints de Categorias: POST, GET, GET /:id, PATCH /:id, DELETE /:id

Lógica: Rotas protegidas. O endpoint GET /categories retorna uma lista inteligente que combina as categorias padrão do sistema (userId: null) com as categorias personalizadas do usuário logado. O sistema impede a criação de categorias duplicadas (mesmo nome e tipo) para um mesmo usuário. Categorias padrão não podem ser editadas ou removidas.

Endpoints de Subcategorias: POST, GET /:id, PATCH /:id, DELETE /:id (em /subcategories) e GET /categories/:id/subcategories

Lógica: CRUD completo e protegido para subcategorias. Um usuário só pode gerenciar subcategorias dentro das categorias às quais ele tem acesso (suas próprias ou as padrão).

5. Módulo de Transações (/transactions)
O núcleo do sistema, responsável por registrar todas as movimentações financeiras.

Endpoint: POST /transactions

Funcionalidade: Registra uma nova transação detalhada.

Lógica: Rota protegida. Aceita um corpo de dados completo, incluindo

movementType, launchType, accountId, categoryId, subcategoryId, amount, dueDate, description, isRecurring, e effectiveDate. A transação é sempre associada ao usuário logado.

Endpoint: GET /transactions

Funcionalidade: Lista as transações do usuário com filtros avançados.

Lógica: Rota protegida. Aceita filtros de query como

?limit=10 para paginação e ?status=pending para buscar apenas transações não efetivadas (effectiveDate: null).

Endpoints: PATCH /:id, DELETE /:id

Funcionalidade: Atualiza e remove uma transação específica.

Lógica: Rotas protegidas que garantem que o usuário só possa modificar suas próprias transações. A regra de negócio que impede a edição ou remoção de uma transação que já foi efetivada (effectiveDate não é nulo) está implementada.

Endpoints: PATCH /:id/pause, PATCH /:id/resume

Funcionalidade: Gerencia o estado de transações recorrentes.

Lógica: Rotas protegidas que permitem pausar ou reativar uma transação marcada como recorrente, alterando o campo isPaused.

6. Módulo de Dashboard (/dashboard)
Fornece dados agregados para uma visão rápida da saúde financeira.

Endpoints: GET /balance, GET /summary-cards, GET /latest-transactions, GET /spending-by-category


Funcionalidade: Agrega e resume os dados financeiros do usuário.

Lógica: Rotas protegidas que realizam cálculos complexos no banco de dados para retornar saldos, totais do mês, últimas transações e gastos percentuais por categoria.

7. Módulo de Relatórios (/reports)
Gera análises financeiras mais detalhadas para o usuário.

Endpoints: GET /daily-summary, GET /monthly, GET /by-category, GET /monthly-full


Funcionalidade: Fornece relatórios financeiros.


Lógica: Rotas protegidas que permitem a geração de resumos diários e relatórios mensais completos (incluindo a categoria de maior gasto), com a opção de filtrar por data.

8. Módulo de IA (/ai)
Fornece funcionalidades de inteligência artificial para auxiliar o usuário.

Endpoint: POST /ai/suggest-category


Funcionalidade: Sugere uma categoria e subcategoria com base na descrição de uma transação.


Lógica: Rota protegida. Atualmente implementada com uma lógica simulada baseada em palavras-chave, pronta para ser substituída por um modelo de IA real no futuro.

9. Módulo de Exportação (/export)
Permite ao usuário ter controle e portabilidade sobre seus dados.

Endpoint: GET /export/csv


Funcionalidade: Gera um arquivo CSV com todas as transações do usuário.

Lógica: Rota protegida. Busca os dados no banco, formata-os no padrão CSV e os envia com os cabeçalhos corretos para forçar o download no cliente.

O sistema deverá contar por padrão global, com categorias e subcategorias de receitas, despesas e investimentos, além de categorias de carteiras e tipo de contas.
O usuário poderá listar todas as categorias e subcategorias. Também poderá criar, editar e excluir, exceto as padrões do sistema.
O serviço de IA deverá catalogar as transações a categoria e subcategoria a qual pertence.

Transações:
Cada transação deverá gerar um código único iniciado por 000001 e deverá conter os seguintes campos/informações:

Data: Preenchimento automático após concluir o lançamento
Tipo de Movimentação: (Entrada ou Saída) - Obrigatório
Conta: Conta de origem do lançamento - Obrigatório
Tipo de Lançamento: Receita, Despesa, Investimento ou Transferência entre contas - Obrigatório
Categoria: Exemplo: Moradia - Obrigatório
Subcategoria: Exemplo: Aluguel - Obrigatório
Valor: 1.200,00 - Obrigatório
Data Vencimento: Para despesas - Obrigatório
Data Recebimento: Para receitas - Obrigatório
Descrição: Texto máximo 50 caracteres - Não obrigatório
Recorrência: Sim ou Não? - Obrigatório
Se Sim, qual o período: Semanal, Quinzenal, Mensal, Trimestral, Semestral ou Anual - Obrigatório se a resposta anterior for sim.
Data de Efetivação: (Data em que ocorreu o Recebimento ou Pagamento) - Obrigatório para efetivar o lançamento (adição ou subtração dos valores em conta), mas não necessariamente no momento do lançamento.

Obs.: Quando o usuário selecionar o tipo de lançamento, a categoria e subcategoria a ser exibida deverá corresponder somente ao tipo de lançamento ou seja, categoria Moradia e Subcategoria nunca deverá ser exibida se o tipo de lançamento for um receita.
